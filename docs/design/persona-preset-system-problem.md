> 기존 설계: [persona-system.md](persona-system.md)
> 다음 단계: `/plan`으로 솔루션 설계

## 핵심 문제

자캐 사용자가 캐릭터의 대사와 반응을 설정하는 과정이 너무 무겁다. 상황 설명을 일일이 쓰거나 모델 매칭에 의존하는 대신, **적은 입력으로 캐릭터다운 대사를 빠르게 채울 수 있는 구조**가 필요하다.

## 핵심 경험

> **사용자가 10분 안에 원하는 페르소나의 초기 버전을 구현할 수 있다.**

## 타겟 사용자

자신의 캐릭터가 어떻게 말하는지 감각이 있는 자캐 소유자. 말투·성격은 머릿속에 있지만, 상황별 대사를 하나하나 쓰는 건 귀찮아하는 사람.

## 시스템 구조 (합의된 방향)

1. **초기 설정**: 프리셋 선택(필수) → 범용 샘플 대사 입력(선택, 최소 0개)
2. **상태 카테고리**: 5개 상태 기반 카테고리 제공
3. **대사 생성**: 프리셋 + 샘플에서 추출한 페르소나 데이터 기반으로 세부 상황별 대사 추천
4. **사용자 액션**: 추천안을 그대로 수락 / 편집 후 사용 / 거절
5. **피드백 루프**: 수락·편집·거절 데이터로 페르소나 데이터 지속 업데이트

## 전제 조건

- 프리셋 + 소수 샘플로 LLM이 캐릭터 말투를 유지한 대사 변형이 가능하다
- 사용자는 자기 캐릭터 말투에 대한 감각이 이미 있다
- 피드백 루프가 페르소나를 유의미하게 개선한다

## 제약사항

- LLM이 완전히 새로운 대사를 마음대로 쓰지 않는다 — 기존 대사/페르소나 기반 변형만
- 초기 설정이 10분을 넘기면 안 된다

## 열린 질문들

- 5개 상태 카테고리의 구체적 목록은?
- 프리셋의 종류와 개수는?
- 페르소나 데이터의 구체적 구조 (어떤 필드를 저장하는지)?
- "대사 변형"의 기술적 구현 방식 (프롬프트 설계 등)?
- 피드백 반영의 구체적 메커니즘?

## 기존 시스템과의 관계

이 시스템은 기존 [persona-system.md](persona-system.md)의 **진화 방향**이다:
- 기존: 씨앗 대사 + 임베딩 매칭 (수동 작성 의존)
- 진화: 프리셋 + 소수 샘플 → LLM 기반 대사 생성 + 피드백 루프

기존 파이프라인의 `OutputProcessor` 슬롯(`PassthroughOutputProcessor` → `LlmVariationOutputProcessor`)이 이 진화의 기술적 접점이 된다.
